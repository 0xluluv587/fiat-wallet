Cursor绘制金融业务产品高保真原型的实践报告

本报告是对Matrix USD金融应用原型设计过程的全面复盘，总结了项目成果、经验教训以及对未来类似项目的建议。通过个人账户与机构账户双层架构的设计和银行账户管理功能的实现，我们探索了AI助手在金融产品设计领域的应用潜力和最佳实践。

一、项目概述

本项目是为Matrix USD应用设计高保真交互原型，重点完成了从机构账户登录到个人/机构账户双层架构的转变，以及完善了OTC法币交易相关的银行账户管理功能。

1.1 主要成果

  • 个人/机构账户双层架构：设计并实现了个人账户登录、机构选择、添加机构和机构认证四个主要页面
  • 银行账户管理功能：改进了银行账户选择页面，设计了银行账户添加页面
  • 整合展示：更新showcase.html以系统展示所有新增功能和页面

1.2 技术实现

所有页面使用HTML、CSS和JavaScript实现，遵循移动端设计规范，采用iPhone 15 Pro的显示框架。代码结构清晰，样式统一，实现了良好的交互体验。

二、成功实践总结

2.1 设计一致性保持

在整个项目中，严格保持了设计语言的一致性，包括色彩系统、组件样式和交互模式，使新增的页面无缝融入现有系统。

项目采用了CSS变量定义全局配色方案，在不同页面保持视觉一致：

:root {
    --primary-color: #0052ff;
    --primary-light: #e9f0ff;
    --primary-dark: #0039c6;
    --neutral-50: #f7f8fa;
    ...
}

表单元素、卡片组件和交互按钮等UI元素均保持一致的样式规范，确保用户在不同页面间切换时体验连贯。

2.2 模块化与组件化思维

采用模块化设计方法，创建了可复用的组件和样式，如表单元素、卡片、上传控件等，大大提高了开发效率和代码可维护性。

例如，在银行账户添加页面中，银行选择组件采用了网格布局的卡片设计，这一模式可以轻松扩展到其他选择类场景：

.bank-option {
    width: calc(33.33% - 16px);
    margin: 0 8px 16px;
    text-align: center;
    cursor: pointer;
}

.bank-option-inner {
    border: 1px solid var(--neutral-200);
    border-radius: 12px;
    padding: 12px 8px;
    transition: all 0.2s;
}

2.3 流程设计的完整性

在设计机构账户流程和银行账户添加流程时，考虑了完整的用户旅程，包括异常情况处理、提示信息和状态反馈，提供了流畅的用户体验。

机构认证页面采用了分步设计，清晰引导用户完成复杂的认证流程：

  1. 第一步：上传企业资质（营业执照、组织机构代码证）
  2. 第二步：填写法人信息并上传证件照
  3. 第三步：确认信息并提交认证申请

同时提供了表单验证、文件上传状态反馈、进度指示器等元素，增强用户信心。

2.4 交互细节的精细处理

注重交互细节，如状态栏时间更新、表单验证反馈、加载状态显示等，提升了产品的专业感和完成度。

例如，在所有页面中都实现了实时时间更新的功能：

function updateStatusBarTime() {
    const now = new Date();
    let hours = now.getHours();
    let minutes = now.getMinutes();
    hours = hours < 10 ? `0${hours}` : hours;
    minutes = minutes < 10 ? `0${minutes}` : minutes;
    document.getElementById('statusBarTime').textContent = `${hours}:${minutes}`;
}

在表单提交时，显示加载状态并禁用按钮，防止重复提交：

submitButton.disabled = true;
submitButton.innerHTML = ' 提交中...';

三、改进空间

3.1 代码复用与提取

在多个页面中存在重复的代码片段，如状态栏时间更新、表单验证逻辑等，这些可以提取为公共函数或组件库，提高维护效率。

建议创建一个共享的JavaScript文件，如common.js，包含通用函数：

// common.js
function updateStatusBarTime() { ... }
function validateForm(formId, rules) { ... }
function handleFileUpload(input, options) { ... }

然后在各页面引入：

<script src="../scripts/common.js"></script>

3.2 响应式设计的增强

当前设计主要针对iPhone尺寸，在不同尺寸的设备上可能需要进一步优化，特别是在极小或极大屏幕上的显示效果。

建议增加更多的媒体查询断点，适应不同尺寸的设备：

@media (max-width: 320px) {
    /* 极小屏幕适配 */
}

@media (min-width: 768px) and (max-width: 1024px) {
    /* 平板适配 */
}

@media (min-width: 1200px) {
    /* 大屏幕适配 */
}

3.3 无障碍性考虑

在页面设计中，对无障碍性（如ARIA标签、键盘导航、对比度等）的考虑较少，这在金融产品中尤为重要。

建议添加必要的无障碍支持：

  • 为表单元素添加适当的aria-*属性
  • 确保所有交互元素可通过键盘操作
  • 检查文本与背景的对比度是否符合WCAG标准
  • 为图像添加alt文本

3.4 数据管理与状态同步

当前页面间的数据传递主要依赖URL参数和sessionStorage，缺乏系统化的状态管理，可能在复杂场景下导致状态同步问题。

建议引入更系统化的状态管理方案，如使用本地存储进行统一的数据处理：

// 定义统一的数据操作接口
const AppState = {
    getUser: () => { return JSON.parse(localStorage.getItem('user') || '{}') },
    setUser: (user) => { localStorage.setItem('user', JSON.stringify(user)) },
    getSelectedOrg: () => { ... },
    setSelectedOrg: (org) => { ... },
    // 其他状态管理函数
};

四、需求方提示词改进建议

4.1 需求精确度

部分需求描述比较概括，如"场景三：OTC法币交易，需要添加一个添加新银行账户的页面"，缺乏具体的字段要求和业务规则描述。

更有效的需求描述可以是：

"场景三：OTC法币交易 - 银行账户添加功能
需要设计一个添加新银行账户的页面，入口在银行账户选择列表页底部。页面需包含以下功能：
1. 银行选择（支持常见银行和"其他"选项）
2. 账户号码输入（需验证格式）
3. 账户持有人姓名（需与实名认证一致）
4. 银行所在地区选择
5. 银行账户证明文件上传
设计风格需与现有页面保持一致，表单提交后显示审核提示。"

4.2 上下文信息提供

在一些需求中，缺乏必要的上下文信息，如业务流程、前后置页面的关系等，导致设计过程中需要多次推断和确认。

建议在提出需求时提供更完整的上下文：

"背景：我们需要从机构账户直接登录改为个人账户登录后选择机构的模式。
业务流程：用户使用邮箱/密码登录个人账户 → 选择要使用的机构账户 → 可添加新机构 → 新机构需完成认证
相关页面：登录页(index.html)、机构选择页(待创建)、机构添加页(待创建)、机构认证页(待创建)
参考设计：可参考Coinbase的组织切换功能..."

4.3 分批迭代的需求提出

需求一次性提出较多页面和功能，可以考虑更有节奏的分批次提出，以保证每个功能都能得到充分的设计和实现。

建议采用迭代式需求提出：

  1. 第一阶段：完成个人账户登录页面改造
  2. 第二阶段：设计机构选择和管理页面
  3. 第三阶段：实现银行账户管理功能
  4. 第四阶段：更新showcase展示系统

五、对AI助手有帮助的核心经验

5.1 项目结构理解策略

在处理复杂项目时，首先使用搜索和浏览功能建立对项目结构的整体认知，再进行针对性的深入研究，这种"广度优先"的方法能提高理解效率。

例如，在本项目中，通过以下步骤快速理解项目：

  1. 使用grep_search查找关键词（如"银行账户"）
  2. 浏览关键文件的结构和主要内容
  3. 理解页面间的导航关系和数据流
  4. 深入研究需要修改的具体文件

5.2 设计模式复用与延续

识别并延续已有的设计模式和代码风格，而非引入新的模式，这样能确保新增功能与现有系统的一致性。

具体策略包括：

  • 分析现有页面的CSS类命名规则和结构
  • 采用相同的JavaScript函数风格和命名约定
  • 复用常见的UI组件结构（如表单、卡片、列表等）
  • 保持交互模式的一致性（如点击、滑动、确认流程等）

5.3 渐进式功能构建

采用渐进式方法构建复杂功能，先实现核心流程，再添加边缘情况处理和交互细节，能有效管理复杂度。

例如，构建银行账户添加页面时：

  1. 第一步：创建基本表单结构和页面布局
  2. 第二步：添加表单验证和提交逻辑
  3. 第三步：实现文件上传功能
  4. 第四步：处理表单错误状态和边缘情况
  5. 第五步：优化交互细节和视觉效果

5.4 任务拆分与解耦策略

将复杂任务拆分为独立的子任务，并确保各部分之间的接口清晰，这样可以更有条理地解决问题。

例如，个人/机构账户架构的实现被拆分为：

  • 改造登录页面（从机构登录改为个人登录）
  • 创建机构选择页面（展示和管理已绑定机构）
  • 设计机构添加页面（加入或创建新机构）
  • 开发机构认证流程（分步骤完成认证）

5.5 Git版本管理最佳实践

遵循清晰的Git提交规范，确保每次提交聚焦于一个主题，并附带描述性的提交信息，有助于项目管理和协作。

在本项目中实践的Git策略：

  • 功能性提交：完整实现一个功能后提交，如"添加个人/机构账户双层架构"
  • 描述性提交信息：详细说明修改内容和目的
  • 分批次提交：避免一次性提交大量更改，便于回溯和管理

六、结论与未来方向

通过本次Matrix USD原型设计项目，我们成功实现了从机构账户登录到个人/机构账户双层架构的转变，并完善了银行账户管理功能。这一实践证明了AI助手在金融产品原型设计中的价值和潜力。

未来的优化方向包括：

  1. 组件库建设：提取和规范化常用UI组件，建立组件库
  2. 响应式设计增强：适配更多设备尺寸和方向
  3. 动效与交互优化：增加适当的过渡动画和交互反馈
  4. 无障碍支持：增强应用的无障碍性和包容性
  5. 自动化测试：引入UI自动化测试，确保视觉和功能一致性

通过总结这些经验和教训，希望能为未来的金融产品原型设计项目提供有价值的参考，并不断优化AI助手在产品设计领域的应用。
